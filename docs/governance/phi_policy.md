# PHI / PII Handling Policy

**Version:** 1.0
**Date:** 2026-03-30
**Scope:** Clinical Note Intelligence Pipeline (this repository)

---

## Summary

This repository was developed and evaluated using only synthetic and de-identified data. No real Protected Health Information (PHI) as defined by HIPAA (45 CFR §160.103) is stored in the repository, transmitted to external services, or included in the evaluation corpus. This section documents the basis for that assertion and the requirements that must be satisfied before the pipeline is used with real patient data.

---

## Dataset-by-Dataset PHI Assessment

### MTSamples (Production Input)

**PHI status: None**

MTSamples is a publicly available collection of medical transcription samples used for training medical transcriptionists. The transcriptions are:
- Created for training purposes and do not correspond to real patient encounters
- Published under CC0 (public domain) with no access restriction
- Not de-identified under HIPAA because they were never PHI to begin with

The dataset contains realistic clinical language but no real patient identifiers, dates of service, or other HIPAA-defined identifiers.

**Conclusion:** Safe to use without PHI controls.

---

### MIMIC-IV Clinical Database Demo (Evaluation Only)

**PHI status: De-identified**

The MIMIC-IV demo dataset is derived from the Beth Israel Deaconess Medical Center EHR and has been de-identified by the MIT Laboratory for Computational Physiology under HIPAA Safe Harbor (45 CFR §164.514(b)):

- Patient names, geographic data below state level, telephone numbers, and all other direct identifiers have been removed
- Dates have been shifted randomly per patient (the relative chronology within a patient record is preserved, but absolute dates are fictional)
- Ages above 89 are reported as 90+ to prevent re-identification of elderly patients
- The dataset is approved for public release by PhysioNet and the BIDMC IRB

Access to the MIMIC-IV demo requires:
1. PhysioNet account creation and training completion
2. Acceptance of the ODbL license terms

**Conclusion:** May be used for research and evaluation. No additional de-identification required. MIMIC data is not stored in this repository (see data lineage).

---

### Synthetic Discharge Notes (Evaluation Corpus)

**PHI status: None — fully synthetic**

The evaluation notes were generated by prompting Claude Haiku 4.5 with structured data from the MIMIC-IV demo (diagnosis descriptions, procedure descriptions, medication names). The generation prompt:
- Provides only coded/labeled clinical entities, not free-text from MIMIC records
- Explicitly prohibits inclusion of ICD codes (which could reference specific patient records)
- Produces novel text that does not reproduce any MIMIC source text

The resulting notes are synthetic clinical documents with no lineage to real patient records beyond the de-identified structured codes.

**Conclusion:** No PHI. Safe for storage, sharing, and training use.

---

### CMS ICD-10-CM Reference Data

**PHI status: None**

ICD-10-CM code tables are clinical terminology standards published by the U.S. government. They contain no patient data.

---

### Anthropic API (Data Processing)

**PHI status: Depends on deployment context**

The pipeline sends clinical note text to the Anthropic API for extraction. In the current implementation:
- Only MTSamples or synthetic evaluation notes are processed
- Neither contains real PHI (see above)

**If the pipeline is extended to process real clinical notes:**

The following requirements apply before any real patient note text is transmitted to the Anthropic API:

1. **Business Associate Agreement (BAA).** Under HIPAA, Anthropic qualifies as a business associate when processing PHI on behalf of a covered entity. A signed BAA must be in place. Verify whether Anthropic offers a BAA for the applicable API tier and contract type.

2. **De-identification.** If a BAA cannot be obtained, note text must be de-identified to Safe Harbor or Expert Determination standard before transmission. This includes removing the 18 HIPAA-defined identifier categories (names, geographic data, dates, phone numbers, etc.).

3. **Data processing terms review.** Review Anthropic's API data usage policies for the applicable contract:
   - Does Anthropic retain input/output for model training? Under what conditions?
   - What is the data retention period for API requests?
   - Is there a zero-data-retention (ZDR) option available?

4. **Minimum necessary standard.** Transmit only the note text required for extraction. Do not include patient metadata (MRN, name, date of birth) in the API payload.

5. **Logging and audit trail.** API calls involving PHI must be logged with sufficient detail to support HIPAA breach notification requirements. Do not log note text content — log note identifiers, timestamps, and API response codes only.

## Repository PHI Controls

The following controls are in place to prevent accidental PHI storage in this repository:

| Control | Implementation |
|---------|---------------|
| Data directory excluded from git | `data/` in `.gitignore` |
| Environment variables for credentials | API keys in `.env` (gitignored); `.env.example` committed without values |
| No patient identifiers in source code | Verified by code review |
| Evaluation results file reviewed | `data/evaluation/evaluation_results.json` contains only MIMIC admission IDs (de-identified integers) and extracted entity strings; no patient names or dates |
| MIMIC data not committed | MIMIC files are researcher-provided and gitignored |

### What Is Committed to Git

| File | Contains PHI? | Basis |
|------|--------------|-------|
| Source code (`src/`) | No | Code only |
| CMS reference files (`data/reference/`) | No | Public domain terminology |
| Evaluation results (`data/evaluation/evaluation_results.json`) | No | De-identified MIMIC IDs + extracted strings |
| Documentation (`docs/`) | No | |
| `.env.example` | No | Template only, no values |

---

## Incident Response

If PHI is accidentally committed to this repository:

1. Do not push to a public remote.
2. Remove the file from git history using `git filter-repo` or `BFG Repo Cleaner`.
3. Rotate any credentials that may have been exposed.
4. Notify the appropriate privacy officer or data steward within 24 hours.
5. Assess whether HIPAA breach notification requirements apply (45 CFR §164.400–414).
